ABSTRACT

This dissertation presents the design, development, and evaluation of UniLearn, a comprehensive web-based Learning Management System (LMS) that addresses the growing demand for accessible, scalable, and feature-rich educational platforms in the post-pandemic digital learning landscape.

UniLearn is built using modern full-stack web technologies, implementing a Model-View-Controller (MVC) architecture with Node.js/Express.js backend, Firebase Firestore NoSQL database, EJS templating for server-side rendering, and Tailwind CSS for responsive UI design. The platform distinguishes itself through seamless integration of Google OAuth 2.0 authentication, Stripe payment processing for subscription tiers, Cloudinary CDN for media management, automated certificate generation, and gamified community features including leaderboards and study groups.

The system supports three distinct user roles—Students, Teachers, and Administrators—with comprehensive role-based access control (RBAC). Core functionality includes course creation and management, multimedia lesson delivery, interactive quizzes with automated grading, real-time progress tracking, and secure payment processing for premium features.

Development followed Agile methodology with continuous integration and deployment via Vercel serverless platform, enabling rapid iteration based on user feedback. Comprehensive testing validated functional requirements, security measures (OWASP Top 10 compliance), and performance benchmarks (100+ concurrent users, <1s response times).

Evaluation results demonstrate successful achievement of all stated objectives, with the platform handling real-world educational scenarios effectively while maintaining security, scalability, and user satisfaction. The project showcases industry-standard development practices including RESTful API design, JWT-based authentication, secure payment integration, and cloud-native deployment strategies.

Keywords: E-Learning, Learning Management System, MVC Architecture, Node.js, Firebase, Full-Stack Development, Educational Technology, Web Application Security, Cloud Computing, RESTful API

CHAPTER 1: INTRODUCTION

1.1 Background and Motivation

The COVID-19 pandemic accelerated digital learning adoption, with UNESCO (2020) reporting 1.6 billion learners affected by school closures. This crisis exposed critical gaps in existing educational technology infrastructure.

Traditional LMS platforms (Moodle, Blackboard, Canvas) suffer from usability challenges and fragmented experiences (Dobre, 2015; Turnbull et al., 2021). Students navigate multiple disconnected tools for content delivery, assessment, and certification—reducing engagement and outcomes (Aldiab et al., 2019). While MOOC platforms (Coursera, Udemy) prioritize user experience, they lack institutional features like role-based access control and comprehensive course management (Yousef et al., 2014).

UniLearn addresses these challenges through modern web technologies, combining intuitive UX, integrated ecosystem (authentication, content, assessment, payment, certification), MVC architecture with RESTful APIs, and cloud-native deployment. 1.2 Problem Statement

**Problem 1: Fragmented User Experience** - Current workflows span multiple disconnected platforms (content hosting, communication, assessment, payment, certification), increasing cognitive load and security vulnerabilities (Mtebe & Kissaka, 2015; Sweller et al., 2011).

**Problem 3: Technical Debt** - Legacy platforms suffer from monolithic codebases, lack of RESTful APIs, session management bottlenecks, and poor mobile responsiveness (Naveh et al., 2012; Crompton & Burke, 2018).

UniLearn addresses these through modern technologies (Node.js, Firebase, EJS), architectural patterns (MVC, RESTful APIs, JWT), and comprehensive feature integration.

2.1 E-Learning Systems Evolution

E-learning evolved through four generations (Aparicio et al., 2016): Computer-Based Training (1960s-80s) with PLATO systems; Web-Based Training (1990s-2000s) introducing Blackboard/Moodle with SCORM standards (ADL, 2004); Social/Mobile Learning (2010s) featuring MOOCs and responsive design (Siemens, 2005); and AI-Enhanced Learning (present) incorporating machine learning and analytics (Zawacki-Richter et al., 2019).

UniLearn implements Third Generation practices (responsive design, social features, cloud deployment) with forward-looking elements (automated certification, gamification).

2.2 Learning Management Systems

LMS platforms facilitate content delivery, assessment, and tracking (Coates et al., 2005). Three categories exist: Open-Source (Moodle, Canvas) with customization but technical complexity (55% market share); Commercial SaaS (Blackboard, D2L) at $5-150/user with vendor lock-in (Grajek, 2019); Consumer Platforms (Udemy, Coursera) prioritizing UX (Shah, 2021).

UniLearn adopts hybrid approach: open-source flexibility with cloud-native scalability, targeting small-medium institutions (Newman, 2015).

2.3 Web Application Architecture Patterns

**MVC**: Separates Models (data), Views (presentation), Controllers (logic) (Krasner & Pope, 1988), reducing bugs by 40% (Garcia et al., 2011). UniLearn uses Firebase models, EJS views, Express controllers.

[Figure 2.1: MVC Architecture Pattern Diagram - TO BE ADDED]

**RESTful APIs**: Stateless, cacheable architecture enabling horizontal scaling (Fielding, 2000). HTTP methods map to CRUD operations. JWT tokens eliminate server-side sessions (Richardson & Ruby, 2007).

[Figure 2.2: RESTful API Request-Response Flow - TO BE ADDED]

2.4 Authentication and Authorization

**OAuth 2.0**: Enables delegated authorization without exposing credentials (RFC 6749, Hardt, 2012). Authorization Code Flow: user authenticates via Google, receives authorization code, exchanges for access token. Benefits include reduced credential proliferation and phishing mitigation (Recordon & Reed, 2006).

[Figure 2.3: OAuth 2.0 Authorization Code Flow - TO BE ADDED]

**JWT**: Compact tokens transmitting claims (RFC 7519, Jones et al., 2015). Three segments: Header, Payload (user ID, role, expiration), Signature. Advantages: stateless, cross-domain compatible, mobile-friendly (Siriwardena, 2014). Security: httpOnly cookies, HTTPS, 24-hour expiration (OWASP, 2021).

[Figure 2.4: JWT Token Authentication Flow - TO BE ADDED]

**RBAC**: Regulates access based on roles (NIST, 2004). UniLearn implements hierarchical roles (Admin > Teacher > Student) with middleware enforcement.

2.5 Payment Gateway Integration

**Stripe**: Developer-friendly APIs supporting 135+ currencies, built-in PCI DSS compliance via hosted Checkout pages, webhooks for event handling, subscription management (Stripe, 2023). Achieves 99.99% uptime and lower abandonment vs. PayPal (Baymard Institute, 2022).

[Figure 2.5: Stripe Payment Processing Architecture - TO BE ADDED]

**PCI DSS Compliance**: UniLearn avoids storing card data (Stripe tokenization), uses hosted Checkout, enforces HTTPS, implements SAQ A (simplest tier), reducing compliance costs from $50K+ to $500 annually (PCI SSC, 2022; Verizon, 2020).

**Subscription Model**: Free tier (3 courses) and Pro ($9.99/month, unlimited). Stripe manages recurring billing, upgrades, retries. Webhooks sync Firestore subscription state (Chargify, 2019).

2.6 Cloud Storage and Content Delivery

**Firebase Firestore**: Document-oriented NoSQL database offering real-time sync, auto-scaling, offline support (Google Cloud, 2023). Flexible JSON-like documents enable rapid prototyping and horizontal scaling (Banker, 2011). Trade-offs: limited queries, cost based on operations (Tudorica & Bucur, 2011). UniLearn optimizes via indexes and denormalized structures.

**Cloudinary CDN**: Distributes assets globally, reducing latency (Peng, 2004). On-the-fly transformations (resize, format conversion), automatic WebP optimization, 25GB free tier (Cloudinary, 2023). Reduces bandwidth 40-60% vs. static JPEGs (HTTP Archive, 2022).

2.7 Web Security Best Practices

**OWASP Top 10**: UniLearn addresses 2021 critical risks (OWASP, 2021): A01 Broken Access Control (RBAC middleware), A02 Cryptographic Failures (bcrypt hashing, HTTPS cookies), A03 Injection (Firestore parameterized queries), A05 Security Misconfiguration (Helmet.js headers), A07 Authentication Failures (JWT expiration, rate limiting).

**Input Validation**: Server-side validation via Joi library prevents type coercion and logic bypasses (OWASP, 2017; Martin & Lam, 2008).

**Output Encoding**: EJS templates auto-escape HTML entities, preventing XSS attacks (OWASP, 2019).

**HTTPS/TLS**: Vercel provides automatic Let's Encrypt certificates (Aas et al., 2019). Helmet.js enforces security headers: Strict-Transport-Security, Content-Security-Policy, X-Frame-Options (Hodges et al., 2012; West et al., 2016).

4.3 Functional Requirements

Functional requirements define specific behaviors and features the system must provide (IEEE, 1998). These are organized by subsystem and prioritized using MoSCoW method (Must have, Should have, Could have, Won't have) (Clegg & Barker, 1994).

FR1: User Authentication and Management (Must Have)

FR1.1: Users must register with email/password or Google OAuth 2.0
FR1.2: System must validate email format and password strength (min 8 characters, alphanumeric)
FR1.3: Users must receive email verification upon registration
FR1.4: System must support password reset via email tokens
FR1.5: Users must be able to update profiles (name, bio, avatar)
FR1.6: System must implement role-based access (Student, Teacher, Admin)

FR2: Course Management (Must Have)

FR2.1: Teachers must create courses with title, description, category, thumbnail
FR2.2: Courses must support draft/published status visibility control
FR2.3: Teachers must add/edit/delete lessons within courses
FR2.4: Lessons must support multiple content types (video, text, code blocks)
FR2.5: System must track lesson completion status per student
FR2.6: Students must browse courses by category and search keywords

FR3: Enrollment and Progress Tracking (Must Have)

FR3.1: Students must enroll in published courses
FR3.2: Free tier users limited to 3 concurrent enrollments
FR3.3: System must calculate course progress (completed lessons / total lessons)
FR3.4: Dashboard must display enrolled courses with progress percentages
FR3.5: Students must resume courses at last accessed lesson

FR4: Quiz and Assessment System (Must Have)

FR4.1: Teachers must create quizzes with multiple question types (MCQ, true/false, short answer)
FR4.2: Quizzes must support time limits and passing scores
FR4.3: System must automatically grade objective questions (MCQ, T/F)
FR4.4: Students must view quiz results with correct/incorrect answers
FR4.5: Teachers must manually grade subjective questions
FR4.6: System must prevent multiple quiz attempts (configurable)

FR5: Payment and Subscription (Must Have)

FR5.1: System must integrate Stripe for payment processing
FR5.2: Users must upgrade from Free to Pro tier ($9.99/month)
FR5.3: Payment flow must redirect to Stripe Checkout
FR5.4: Webhooks must update subscription status automatically
FR5.5: Pro users must have unlimited course enrollments

FR6: Certificate Generation (Should Have)

FR6.1: System must generate PDF certificates upon 100% course completion
FR6.2: Certificates must include student name, course title, completion date, unique ID

5.2.2 Database Relationships and Design Rationale

One-to-Many Relationships:
User → Courses (1 teacher creates many courses)
Course → Lessons (1 course contains many lessons)
Course → Quizzes (1 course has multiple assessments)
Quiz → Questions (1 quiz contains many questions)
User → Enrollments (1 student enrolls in many courses)
User → Quiz Attempts (1 student attempts multiple quizzes)
Study Group → Posts (1 group contains many discussion posts)
Post → Replies (1 post receives multiple replies)

Many-to-Many Relationships (Handled via Join Collections):
Users ↔ Study Groups (via study_groups.memberIds array)
Users ↔ Courses (via enrollments collection)
Users ↔ Achievements (via user_achievements collection)

Denormalization Strategy Justification:

User Names/Avatars in Posts: Reduces 2-3 additional reads per post display; acceptable staleness (users rarely change names)
Teacher Name in Courses: Course listings require teacher info; denormalization reduces course display from 2 reads to 1
Enrollment Counts: Heavily read, rarely written; cache invalidation via Firestore transactions on enroll/unenroll
Achievement Metadata in User Achievements: Prevents repeated achievement document reads when displaying user profiles
Leaderboard Ranks: Precomputed via scheduled Cloud Function (daily); real-time ranking would require sorting millions of documents

Database Indexes for Query Optimization:
courses: Composite index on (published, category, createdAt) - enables published course filtering by category with recent-first sorting
enrollments: Composite index on (userId, courseId) - fast duplicate enrollment checks and user's course lookups
lessons: Index on (courseId, order) - sequential lesson retrieval for course progression
quiz_attempts: Index on (userId, quizId, submittedAt) - enables quiz history retrieval sorted by submission time
study_groups: Index on (isPublic, memberCount) - discovery of popular public groups
group_posts: Composite index on (groupId, createdAt) - chronological post feeds per group
leaderboard: Index on totalPoints DESC - global leaderboard ranking queries
user_achievements: Composite index on (userId, unlockedAt) - user achievement timeline
activity_log: Composite index on (userId, createdAt DESC) - user activity feeds

Firestore Query Limitations and Workarounds:

Firestore lacks full-text search; course search implemented via:
Client-side filtering for small datasets (<100 courses)
Third-party search integration (Algolia) planned for scale (v2.0)

Complex aggregations (e.g., average quiz scores across platform) handled via:

const authMiddleware = async (req, res, next) => {
  const token = req.cookies.token || req.headers.authorization?.split(' ')[1];

  if (!token) {

5.5.6 Firestore Security Rules (Extended)

Complete Security Rules:
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Helper functions
    function isAuthenticated() {
      return request.auth != null;
    }

    function isOwner(userId) {
      return request.auth.uid == userId;
    }

    function hasRole(role) {
      return request.auth.token.role == role;
    }

    // Users collection
    match /users/{userId} {
      allow read: if isAuthenticated();
      allow write: if isOwner(userId);
      allow delete: if hasRole('admin');
    }

    // Courses collection
    match /courses/{courseId} {
      allow read: if resource.data.published == true || isOwner(resource.data.teacherId) || hasRole('admin');
      allow create: if isAuthenticated() && (hasRole('teacher') || hasRole('admin'));
      allow update: if isOwner(resource.data.teacherId) || hasRole('admin');
      allow delete: if isOwner(resource.data.teacherId) || hasRole('admin');
    }

    // Enrollments collection
    match /enrollments/{enrollmentId} {
      allow read: if isOwner(resource.data.userId) || hasRole('admin');
      allow create: if isAuthenticated() && isOwner(request.resource.data.userId);
      allow update: if isOwner(resource.data.userId); // Progress updates
      allow delete: if isOwner(resource.data.userId) || hasRole('admin');
    }

    // Quizzes collection
    match /quizzes/{quizId} {
      allow read: if isAuthenticated(); // Students need to see quizzes
      allow create: if hasRole('teacher') || hasRole('admin');
      allow update, delete: if isOwner(resource.data.createdBy) || hasRole('admin');
    }

    // Quiz attempts collection
    match /quiz_attempts/{attemptId} {
      allow read: if isOwner(resource.data.userId) || hasRole('teacher') || hasRole('admin');
      allow create: if isAuthenticated() && isOwner(request.resource.data.userId);
      allow update: if false; // Immutable after creation
      allow delete: if hasRole('admin');
    }

    // Study groups collection
    match /study_groups/{groupId} {
      allow read: if resource.data.isPublic == true || resource.data.memberIds.hasAny([request.auth.uid]);
      allow create: if isAuthenticated();
      allow update: if isOwner(resource.data.createdBy) || hasRole('admin');
      allow delete: if isOwner(resource.data.createdBy) || hasRole('admin');
    }

    // Leaderboard collection
    match /leaderboard/{userId} {
      allow read: if isAuthenticated(); // Public leaderboard
      allow write: if false; // Only backend can update (via Admin SDK)
    }

    // Certificates collection
    match /certificates/{certificateId} {
      allow read: if isOwner(resource.data.userId) || hasRole('admin');
      allow write: if false; // Only backend generates certificates
    }

    // Orders collection
    match /orders/{orderId} {
      allow read: if isOwner(resource.data.userId) || hasRole('admin');
      allow write: if false; // Only Stripe webhooks create orders
    }

    // Admin logs collection
    match /admin_logs/{logId} {
      allow read: if hasRole('admin');
      allow write: if false; // Only backend writes logs
    }
  }
}

Security Rules Benefits:
Defense-in-depth: Database-level security even if application logic fails
Zero-trust architecture: Every request authenticated and authorized
Granular permissions: Different rules per collection and operation
Performance: Firestore evaluates rules server-side (no application overhead)

exports.handleWebhook = async (req, res) => {
  const sig = req.headers['stripe-signature'];
  let event;

  try {
    event = stripe.webhooks.constructEvent(
      req.body,
      sig,
      process.env.STRIPE_WEBHOOK_SECRET
    );
  } catch (err) {
    return res.status(400).send(`Webhook Error: ${err.message}`);
  }

  if (event.type === 'checkout.session.completed') {
    const session = event.data.object;

    // Update order status
    await Order.updateBySessionId(session.id, { status: 'completed' });

    // Upgrade user subscription
    await User.update(session.client_reference_id, {
      subscriptionTier: 'pro'
    });
  }

  res.json({ received: true });
};

**Process**: Iterative testing (3 UAT cycles) catches issues early; concurrent documentation saves time (20+ hours); scope discipline prevents feature creep, ensuring quality over quantity.
